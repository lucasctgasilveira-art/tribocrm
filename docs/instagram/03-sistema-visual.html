01 Princípios do sistema

As regras que mantêm o sistema coerente conforme a quantidade de peças cresce.

1
tokens.css é fonte única de verdade. Nenhum template hard-codeia cor ou tipografia.
2
Cada template é standalone. Abre direto no navegador, sem build step.
3
Conteúdo é editável via data-edit. Pipeline lê essas chaves e substitui.
4
Símbolo D + assinatura sempre presentes. Convenção de marca não-negociável.
5
DM Sans é a única tipografia. Pesos 400/500/600/700/800 — nada além disso.
6
Paleta: preto #0a0a0a · laranja #f97316 · branco #ffffff. Sem cinzas além de transparências.

04 Convenção data-edit

Cada ponto editável é marcado com data-edit="<chave>". Pipeline lê e substitui.

No template HTML:

<h1 class="t-h1" data-edit="titulo">Os 4 momentos do funil...</h1>
<span class="brand-signature" data-edit="handle">@tribocrmoficial</span>

No YAML de pauta (Bloco 4 vai consumir):

template: carrossel-metodo-capa
data:
  pilar: "Método"
  titulo: "Os 4 momentos do funil que sua PME tá pulando"
  slide-number: "01 / 08"
  handle: "@tribocrmoficial"

O pipeline injeta cada valor no innerHTML ou textContent do elemento marcado, antes de capturar o screenshot.

05 Como vira pipeline (preview do Bloco 4)

O sistema visual ficou pronto pro próximo bloco consumir.

[ posts/2026-05-04-funil-capa.yaml ]
            ↓
       template:
       carrossel-metodo-capa.html
            ↓
       data-edit substitution
            ↓
       Puppeteer · headless Chrome
            ↓
       capture .canvas @ 1080×1350
            ↓
       output/2026-05-04-funil-capa.png

CLI alvo:

npm run post -- --pauta posts/2026-05-04-funil-capa.yaml
Bloco 4 vai entregar: package.json com Puppeteer + js-yaml · script Node scripts/render-post.mjs · estrutura posts/ e output/ · auto-detecção de tamanho via data-format · workflow ponta-a-ponta funcionando com 1+ peça do calendário Mês 1.
